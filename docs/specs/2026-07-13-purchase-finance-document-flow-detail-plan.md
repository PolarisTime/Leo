# 采购财务逐行单据流明细改造计划

## 1. 文档状态

- 状态：业务规则已确认，实施中
- 日期：2026-07-13
- 涉及仓库：后端 `leo`、前端 `aries`
- 当前范围：采购订单、采购入库、供应商对账、采购退款及采购收票的逐行追溯，以及付款、收款的单据总额资金追溯
- 查询根范围：`结算主体 ID + 供应商 ID`
- 实施约束：本文档只定义业务子账和单据流读模型，不引入会计科目、复式记账、总账凭证或会计期间

### 1.1 与既有计划的关系

现有 [采购入库拆分与采购结算记账重构计划](2026-07-13-purchase-inbound-settlement-ledger-redesign-plan.md)
包含“删除采购退款单和供应商退款到账单”的旧方案，与本轮已确认需求冲突。

发生冲突时，以本文件关于以下内容的决策为准：

1. 保留采购退款单，用于逐行确认未提货退款金额；
2. 供应商实际退款统一使用通用收款单，不再保留独立的供应商退款收款业务模型；
3. 保留采购订单、采购入库等业务单据，不使用单一泛化事件替代用户可追溯的原始单据；
4. 新增统一逐行单据流读模型，但不得把它当作第二套可编辑业务账本；
5. 采购入库自动拆分、过磅规则和结单入口仍可复用既有计划中不冲突的部分。

在本文件完成全部业务确认前，不得删除采购退款业务；独立供应商退款到账模块只有在数据完整迁入通用收款单后才能移除。

## 2. 问题定义

当前应收应付明细可以展示采购入库、付款和供应商退款到账等单据级记录，但无法满足逐行追溯：

1. 采购入库在查询中按入库单汇总，丢失真实入库明细行；
2. 现有明细响应缺少单据行号、来源行号、物料、数量、实际重量和单价；
3. 付款和收款只影响“结算主体 + 供应商”的总额余额，不应被强制拆分或核销到采购订单、入库、退款等业务明细；
4. 现有通用收款单写死客户和客户项目，不能表达结算主体收到供应商退款；独立供应商退款到账单又只有表头金额；
5. 采购订单、入库、对账、付款属于同一业务的不同阶段，若直接汇总所有行金额会重复计算；
6. 现有借方、贷方字段是业务子账方向，不能直接翻译为用户所需的现金“支出、收入”。

## 3. 目标

1. 在同一张只读表中逐行展示业务单据明细，并为每张付款单、收款单各展示一条表头总额资金行；
2. 业务明细行展示当前单号/行号与直接来源单号/行号；资金行使用自身资金单据追溯，不关联采购业务明细；
3. 采购入库逐行展示实际过磅重量、单价和结算行金额；
4. 每张采购付款单只展示一条付款总额行，并且只在“支出”列记现金流出；
5. 每张供应商类型收款单只展示一条收款总额行，并且只在“收入”列记现金流入；
6. 采购退款作为独立业务行展示，不用负数覆盖采购订单或入库行；
7. 分别汇总计划金额、实际入库结算、确认退款、现金支出、现金收入、净现金支出和历史调整审计金额；
8. 业务行可以追溯到稳定的来源单据明细 ID，资金行可以追溯到稳定的付款单或收款单 ID；历史台账调整保留原单号、方向、影响和金额，但不再参与余额。

## 4. 非目标

1. 不允许用户直接新增、编辑或删除统一单据流行；
2. 不把统一单据流表作为业务单据之外的第二套事实来源；
3. 不对采购订单、采购入库、对账和付款的行金额做无口径的总计；
4. 业务单据和原始资金单据金额保持非负，方向由行角色和支出/收入列表示；只有资金冲销行允许在被冲销的原方向列显示负数；
5. 不在本阶段建设法定总账、会计凭证、税务申报或多币种汇兑；
6. 不把付款或收款按比例分摊到采购业务明细，也不建立并不存在的订单行级资金核销关系。

## 5. 已确认业务决策

1. 采购订单表达采购计划，不直接计入现金支出；
2. 采购入库表达实际到货和结算金额，不直接表示已经付款；
3. 采购付款单审核后，才在支出列产生金额；审核表示资金已经实际付出；
4. 采购退款单表达已经确认的应退金额，不直接表示资金到账；
5. 供应商退款统一使用收款单，收款单审核后才在收入列产生金额；审核表示资金已经实际收到；
6. 采购退款行和供应商类型收款行必须独立追加，不能覆盖或负数冲减原业务行；
7. 采购订单、入库、对账、退款和收票有多少条真实明细就展示多少行；付款单和收款单属于例外，每张资金单据只展示一条表头总额行；
8. 采购订单和采购入库可以同时展示以对比计划与实际，但不能重复进入现金汇总；
9. 过磅商品的采购入库行以实际过磅重量计算结算金额；
10. 未提货退款按采购订单来源明细逐行生成和追溯；
11. 表格固定包含“行金额、支出、收入”三个不同口径的金额列。
12. “完成采购”表示采购履约已经结束，不等待供应商退款实际到账；
13. 完成采购时如果存在未入库剩余数量，后端必须在同一事务内按来源明细自动生成采购退款单；
14. 自动生成的采购退款单状态为“草稿”，未审核前不产生正式退款财务影响；
15. 财务核对通过后把采购退款单改为“已审核”，后续实际到账由通用收款单按收款单总额确认；
16. 采购退款单只保留“草稿、已审核”两个业务状态；“未结清、部分结清、已结清”必须根据账簿汇总余额实时计算，不能保存为退款单状态；
17. 采购订单完成后不额外维护“待退款审核、待退款到账、已结清”等财务状态字段，页面通过账簿汇总结果展示结算进度；
18. 是否结清只按同一结算主体与同一供应商的净余额判断，不要求每张采购订单或采购退款单分别结清；
19. 所有对供应商的资金流出使用付款单，所有从供应商收到的资金使用收款单，采购退款单本身不伪装成收款事实；
20. 任何影响供应商余额的业务或资金变化都必须由正式单据产生，禁止绕过单据直接修改账簿余额；
21. 台账调整写入口停用；历史调整只保留只读审计，不允许新增、修改、审核或删除；
22. 付款单和收款单只对应同一结算主体与供应商之间的总额资金，不关联采购订单、入库、对账、退款或收票明细；
23. 每张有效付款单只生成一条 `CASH_OUT` 总额行，每张有效收款单只生成一条 `CASH_IN` 总额行，资金行的上游来源单号/行号为空；
24. 供应商账簿余额由后端实时计算并保持只读；允许通过多张付款单或收款单部分结算，经办人只填写本张资金单总额，剩余余额由系统自动计算；
25. 单据流默认按业务发生时间升序展示；相同业务日期按审核或生效时间、稳定 ID 和行号确定唯一顺序，再按最终结果生成流转序号。
26. 单据流默认同时展示未生效和已生效行；未达到各单据生效状态的行必须明确标记为“未生效”，且不得参与计划、结算、退款、支出、收入、余额等任何有效汇总。
27. 供应商对账单和采购收票单默认加入统一单据流；两者分别使用对账与收票口径独立汇总，不计入支出、收入或净现金支出，也不得重复计入采购入库结算。
28. 统一单据流固定按“结算主体 ID + 供应商 ID”查询；采购订单 ID 只能作为可选筛选或跳转参数，不能决定付款、收款的归属或截断供应商资金记录。
29. 付款金额允许超过当前应付余额，超出部分形成同一“结算主体 + 供应商”下的预付款余额；供应商退回预付款时使用收款单记录，禁止直接冲改预付款余额。
30. 后续采购入库审核产生新应付时，系统按供应商总账净额自动抵扣已有预付款，不生成预付款抵扣单，也不建立预付款与采购订单或入库明细的分配关系。
31. “供应商预付款退款”用途的收款单不得超过审核时的预付款余额；实际到账超过该余额时，超出部分必须拆为“供应商其他收款”用途的独立收款单，禁止把超额金额伪装成预付款退款。
32. 付款单和收款单统一只保留“草稿、已审核”两个状态；审核即确认资金已实际发生并立即影响支出、收入和供应商账簿，银行资金尚未发生时不得提前审核。
33. 已审核付款单、收款单禁止反审核、修改和删除；填错时必须创建关联原资金单的冲销单，冲销审核后再创建正确单据。供应商实际退回资金仍使用收款单，不得用冲销单代替真实资金流。
34. 资金冲销单自身的行金额保持正数；付款冲销在“支出”列记负数，收款冲销在“收入”列记负数，相反方向列为空，禁止把账务冲销伪装成真实反向收付款。
35. 供应商对账单只承担金额核对和确认职责，不重复增加应付、不产生现金收支；只有根采购订单处于“完成采购”状态的已审核采购入库明细，才能进入供应商对账候选。
36. 同一结算主体、同一供应商的多张“完成采购”订单可以合并生成一张供应商对账单；每条对账明细必须保留原采购入库明细及根采购订单明细，不允许跨结算主体或跨供应商混合对账。
37. 供应商对账草稿按每条来源明细的全部剩余可对账金额预占容量；其他草稿不能重复选择，修改或删除草稿释放预占，审核后预占原子转换为正式已对账金额。
38. 同一采购入库明细不允许分批对账；选择来源后，本次对账金额必须由后端取全部剩余可对账金额并保持只读，不能由用户减少或拆到多张新对账单。

## 6. 统一表格定义

### 6.1 用户可见列

| 列 | 定义 | 汇总规则 |
| --- | --- | --- |
| 流转序号 | 按最终排序结果使用 `row_number()` 生成，不作为业务主键 | 不汇总 |
| 单据类型 | 采购订单、采购入库单、供应商对账单、付款单、采购退款单、收款单、采购收票单、历史台账调整单 | 分组筛选 |
| 单号 | 当前来源单据编号 | 不汇总 |
| 行号 | 业务单据的真实明细行号；资金总额行固定显示 `1` | 不汇总 |
| 来源单号/行号 | 业务行的直接来源；付款、收款资金行显示 `-` | 不汇总 |
| 物料 | 当前业务行快照；付款、收款资金行为空 | 不汇总 |
| 数量 | 当前业务行数量；付款、收款资金行为空 | 按同一单据角色汇总 |
| 实际重量 | 采购入库优先取过磅重量；非过磅行按业务需要为空 | 按同一单据角色汇总 |
| 单价 | 当前来源行采用的单价 | 不直接汇总 |
| 行金额 | 业务明细金额；资金行取付款单或收款单表头总额 | 只能按单据角色分别汇总 |
| 支出 | 已审核付款单总额；付款冲销在本列记负数 | 按有符号金额汇总 |
| 收入 | 已审核收款单总额；收款冲销在本列记负数 | 按有符号金额汇总 |
| 历史调整方向 | 仅历史台账调整行显示原“应收/应付”方向 | 只审计，不计入余额 |
| 历史余额影响 | 仅历史台账调整行显示原“增加余额/减少余额” | 只审计，不计入余额 |

`行金额`建议在页面显示名称中使用“单据行金额”，避免用户误认为所有行可以直接相加。

## 7. 单据行映射

| 单据类型 | 行角色 | 行金额口径 | 支出 | 收入 | 直接来源 |
| --- | --- | --- | --- | --- | --- |
| 采购订单行 | `PLAN` | 订单明细金额 | 空 | 空 | 无或采购合同明细 |
| 采购入库行 | `RECOGNITION` | `amount + weightAdjustmentAmount` | 空 | 空 | 采购订单明细 |
| 供应商对账行 | `RECONCILIATION` | 对账明细确认金额 | 空 | 空 | 已完成采购订单下的采购入库明细 |
| 采购退款行 | `REFUND_CLAIM` | 已确认应退明细金额 | 空 | 空 | 采购订单明细 |
| 付款单总额行 | `CASH_OUT` | 付款单表头总额 | 同行金额 | 空 | 无；资金单据自身即余额变动凭证 |
| 收款单总额行 | `CASH_IN` | 收款单表头总额 | 空 | 同行金额 | 无；资金单据自身即余额变动凭证 |
| 付款冲销行 | `REVERSAL` | 正数的本次冲销金额 | 负数的本次冲销金额 | 空 | 被冲销的付款单 |
| 收款冲销行 | `REVERSAL` | 正数的本次冲销金额 | 空 | 负数的本次冲销金额 | 被冲销的收款单 |
| 采购收票行 | `INVOICE` | 发票明细金额 | 空 | 空 | 采购入库或采购订单明细 |
| 历史台账调整行 | `HISTORICAL_AUDIT` | 原调整金额 | 空 | 空 | 原调整类型；只读审计，不参与余额 |

不论上游存在多少条采购业务明细，每张付款单、收款单都只追加一条资金总额行。资金行不继承物料、数量、重量、单价或
业务来源明细，也不进行订单行级核销；它只通过资金单据 ID、结算主体 ID、供应商 ID 和金额参与供应商总额账簿计算。

冲销行是唯一允许在支出或收入列出现负数的行。冲销单自身金额仍为正数，负号仅表达对原资金方向的撤销；
付款冲销不得记成收入，收款冲销不得记成支出。

## 8. 状态与生效口径

1. 草稿行默认在单据流中显示，但必须使用明确的“未生效”状态样式，并且不参与任何金额汇总；
2. 采购订单只有进入“完成采购”后才进入有效计划金额；“已审核”订单仍展示，但不进入财务有效汇总；
3. 采购入库只有已审核或完成入库，且根采购订单已完成采购后，才进入实际入库结算；
4. 根采购订单完成采购后，其已审核入库明细才允许创建供应商对账；对账审核后进入已对账视图，但不再次改变应付余额；
5. 采购退款单只允许“草稿 -> 已审核”和合法反向流转，审核后才进入确认退款金额；
6. 付款单审核后才产生支出，审核前必须确认资金已经实际付出；
7. 收款单审核后才产生收入，审核前必须确认资金已经实际收到；
8. 已审核资金单据禁止反审核、修改和删除；已审核冲销单产生反向账簿影响，但原资金行和原审核日志必须完整保留；
9. 软删除单据默认不展示，也不参与汇总；审计查询可提供单独入口。
10. 历史台账调整行以原状态标记是否已审核，审核金额只进入“历史调整审计金额”；无论状态如何都不参与应付、预付款、支出或收入。

### 8.1 完成采购与自动退款

目标流程：

```text
审核最后一张采购入库单并确认完成采购
  -> 后端锁定采购订单及相关明细
  -> 计算每条采购订单明细的剩余未入库数量和退款金额
  -> 剩余数量大于零时逐行生成采购退款单
  -> 采购订单进入“完成采购”
  -> 采购退款单以“草稿”状态等待财务核对
  -> 财务核对通过后进入“已审核”，退款生效
  -> 需要供应商实际退回资金时创建供应商类型的收款单
```

事务要求：

1. 采购订单完成和采购退款单生成必须处于同一数据库事务；
2. 任一退款明细计算、保存或容量校验失败时，完成采购整体回滚；
3. 没有剩余数量时直接完成采购，不生成零金额或空退款单；
4. 使用稳定幂等键防止重复点击生成多张活动退款单；
5. 退款数量、来源采购订单明细、供应商和结算主体由后端生成，财务不得换来源；
6. 财务核对未通过时应驳回并记录原因，不得直接改写系统计算的剩余数量；
7. 采购可以已经“完成采购”，但退款单仍可处于草稿；退款进度和是否结清从账簿金额计算，不新增持久化财务状态。

## 9. 汇总口径

页面不得提供“全部行金额合计”。顶部或表尾分别展示：

```text
采购计划金额       = sum(PLAN.lineAmount)
实际入库结算       = sum(RECOGNITION.lineAmount)
已对账确认金额     = sum(RECONCILIATION.lineAmount)
确认退款金额       = sum(REFUND_CLAIM.lineAmount)
已收票金额         = sum(INVOICE.lineAmount)
历史调整审计金额   = sum(HISTORICAL_AUDIT.lineAmount where effective)
实际支出金额       = sum(expenseAmount)
实际收入金额       = sum(incomeAmount)
净现金支出         = 实际支出金额 - 实际收入金额
```

`expenseAmount` 和 `incomeAmount` 的汇总包含对应冲销负数，因此结果表示有效净支出和有效净收入；原始资金总额与冲销总额应另作审计统计，不能混入业务行金额总计。

`historicalAdjustmentAmount` 仅用于核对停用前的历史调整单，不进入 `netPosition`、应付余额、预付款余额或现金汇总。缺少稳定结算主体的旧调整仍可在停用后的台账调整只读查询中审计，但不会被猜测归入任一供应商财务流。

应付、应收退款和预存余额继续由业务子账按供应商与结算主体计算，不能只用统一表中不同阶段的行金额相减。

### 9.1 供应商净额结算

采购结算边界固定为：

```text
settlementCompanyId + supplierId
```

供应商名称和结算主体名称只保存快照，不参与身份判断。同一供应商在不同结算主体下不得互相抵扣。

以正数表示结算主体仍应付供应商，以负数表示供应商应退结算主体：

```text
netPosition
= 已完成采购订单下的已审核采购入库结算金额
- 有效付款金额
+ 有效收款金额
```

```text
netPosition > 0  -> 结算主体仍应付供应商
netPosition < 0  -> 结算主体在供应商处存在预付款余额
netPosition = 0  -> 双方金额已结清
```

页面使用互斥的只读金额展示余额方向：

```text
payableBalance    = max(netPosition, 0)
prepaymentBalance = max(-netPosition, 0)
```

付款超过 `payableBalance` 时，超过部分自动进入 `prepaymentBalance`。供应商实际退回预付款必须创建收款单，
收款单生效后增加 `netPosition`、减少预付款余额，不允许用户直接编辑预付款余额。

后续采购入库审核增加 `netPosition` 时，会自然减少 `prepaymentBalance`；超过预付款的部分转为 `payableBalance`。
该过程是同一供应商账簿的实时净额计算，不创建抵扣单，也不保存订单级抵扣关系。

“供应商预付款退款”收款审核时必须锁定同一结算主体与供应商的账簿并重新计算 `prepaymentBalance`，
收款金额不得超过该余额。实际多到账部分使用独立的“供应商其他收款”单据和用途记录，不能静默改变原退款用途。

采购退款单用于确认未提货和应退业务依据，不重复改变已经按实际采购入库确认的应付金额。付款和收款才表达真实资金流；
付款和收款均以资金单据表头总额进入供应商账簿，不分配、不核销到任何采购业务明细。资金行的审计依据是付款单或收款单自身，
供应商是否结清只按结算主体与供应商的净余额判断。

采购退款单只展示其确认金额，不展示“本退款单已收款金额”或单据级结清结果，因为收款不对应某张退款单。
唯一结清依据是同一结算主体与供应商的 `netPosition`。该结果是账簿计算字段，不是采购订单、采购退款单、付款单或收款单状态，
也不能由用户直接修改。

使用已讨论示例：

```text
采购计划金额：60,500
实际入库结算：54,120
确认退款金额： 7,200
实际支出金额：60,500
实际收入金额： 7,200
净现金支出：  53,300
```

示例中计划金额、入库金额和退款金额属于不同业务口径，只能分别展示。现金口径只使用支出与收入。

## 10. 业务来源与资金凭证追溯规则

1. 业务明细行展示直接来源，查询和分组同时保存根采购订单明细；
2. 采购入库行直接关联采购订单明细；
3. 供应商对账行直接关联采购入库明细，且该入库明细的根采购订单必须处于“完成采购”；
4. 采购退款行直接关联采购订单明细；
5. 采购收票行直接关联采购入库明细或采购订单明细；
6. 付款和收款资金行不填写上游业务来源，`documentId`、`documentNo` 直接指向产生余额变化的付款单或收款单；
7. 资金单据的结算主体与供应商必须与被查询的供应商账簿维度一致；
8. 台账调整写入口已停用；可稳定归属的历史调整在统一流中以 `HISTORICAL_AUDIT` 只读展示，不允许继续改变供应商余额；
9. 任何余额变化都必须能追溯到有效业务单据或资金单据，禁止直接插入或修改账簿余额；
10. 付款或收款生效后，结算主体、供应商和金额保持只读；需要纠错时必须创建指向原资金单据的冲销单；
11. 冲销单必须保存原资金单据 ID、冲销金额和原因，同一原单累计有效冲销金额不得超过原单有效金额；
12. 业务来源明细被有效下游业务单据引用后，上游变更必须经过现有下游保护，不能静默换来源。

同一供应商对账单可以包含多张采购订单和多张采购入库单，但每一条对账行只能指向一个稳定的采购入库明细 ID，
并保留对应根采购订单明细 ID。对账单表头的结算主体与供应商由候选明细共同确定，后续加入的明细必须与表头完全一致。

## 11. 当前代码缺口

### 11.1 应收应付查询粒度不足

`ReceivablePayableQueryRepository` 当前在 `inbound_payable` 中按 `inbound_id` 汇总采购入库金额，随后把
`source_line_id` 固定为 `NULL`。必须改为逐个 `po_purchase_inbound_item` 生成明细读模型行。

### 11.2 明细响应字段不足

`ReceivablePayableDetailItemResponse` 当前只有单据级来源、日期、借贷金额和余额。需要新增本计划第 6.2 节的行级字段，
或者新增独立 `PurchaseDocumentFlowLineResponse`，避免把应收应付汇总 DTO 继续扩成混合职责对象。

推荐新增独立响应对象和查询服务，不修改现有应收应付汇总接口的既有语义。

### 11.3 供应商对账按金额预占

每条已审核采购入库明细的可对账金额由后端实时计算：

```text
availableReconciliationAmount
= inboundSettlementAmount
- auditedReconciliationAmount
- activeDraftReservedAmount
```

约束：

1. 草稿保存、修改、删除和审核必须锁定全部来源入库明细，并在同一事务内更新预占结果；
2. 多来源批量操作按稳定 ID 排序加锁，避免并发保存时产生死锁；
3. 本次对账金额必须大于零且等于扣除其他活动草稿后的全部剩余可对账金额，由后端计算并保持只读；
4. 修改草稿时先排除当前草稿自身原预占，再校验新金额，禁止重复扣减；
5. 删除草稿立即释放预占，审核时把预占原子转换为正式已对账金额；
6. 草稿预占不改变应付余额、支出或收入，也不进入已对账金额汇总；
7. 候选查询和保存命令必须复用同一容量口径，前端传入的剩余金额只作展示，不能作为审核依据。

### 11.4 付款需要按单据总额进入读模型

`fm_payment_allocation` 当前可能保留供应商或物流对账单表头关联，但统一单据流不得把该关联解释为采购订单行级核销，
也不新增 `fm_payment_allocation_item`。每张付款单按最终有效表头金额生成一条 `CASH_OUT` 行。

约束：

1. 付款金额必须大于零；
2. 结算主体和供应商不能为空，并作为供应商账簿归属维度；
3. 付款单审核后才改变账簿余额并进入支出汇总，审核即确认资金已实际付出；
4. 已审核后禁止直接修改结算主体、供应商和金额，纠错必须走正式反向流程；
5. 读模型中的业务来源字段、物料、数量、重量和单价均为空；
6. 付款单允许超过当前应付余额，超过部分按供应商账簿实时计算为预付款余额。

### 11.5 通用收款单不支持供应商

当前 `fm_receipt`、请求对象、身份解析器和前端页面均以客户及客户项目为固定模型。需将收款单泛化为往来方模型：

```text
business_type
counterparty_type
counterparty_id
counterparty_code
counterparty_name
receipt_purpose
settlement_company_id
settlement_company_name
```

收款用途至少支持客户对账收款和供应商资金收款。供应商收款时客户项目为空，也不关联采购退款单或退款明细。
现有客户收款分配模型可以在销售边界内继续使用，但不得为了供应商收款扩展采购退款来源分配字段。

供应商资金收款用途至少细分为：

```text
SUPPLIER_PREPAYMENT_REFUND
SUPPLIER_OTHER_RECEIPT
```

约束：

1. 供应商类型收款的金额必须大于零；
2. 结算主体和供应商不能为空，并作为供应商账簿归属维度；
3. 收款单审核后才改变账簿余额并进入收入汇总，审核即确认资金已实际收到；
4. 已审核后禁止直接修改往来方、结算主体和金额，纠错必须走正式反向流程；
5. 每张供应商类型收款单只生成一条 `CASH_IN` 总额行；
6. `SUPPLIER_PREPAYMENT_REFUND` 审核时必须锁定并校验预付款余额，收款金额不得超过该余额；
7. 超过预付款余额的实际到账必须拆为 `SUPPLIER_OTHER_RECEIPT`，不得自动合并或更改用途；
8. 现有 `fm_supplier_refund_receipt` 历史数据迁移为供应商类型的通用收款单表头记录，验证完成后再移除独立模块。

### 11.6 资金冲销约束

1. 冲销单只保留“草稿、已审核”两个状态；
2. 草稿冲销不影响账簿，审核后才产生与原资金单相反的余额影响；
3. 审核时必须锁定原资金单并重新计算可冲销余额，禁止并发超额冲销；
4. 冲销单的结算主体和供应商由原单继承且不可修改；
5. 冲销金额必须大于零且不超过原单剩余可冲销金额；
6. 原资金单和已审核冲销单均不得物理删除；
7. 冲销原因必填，冲销行直接来源固定为原付款单或原收款单；
8. 冲销只纠正系统资金记录，银行实际发生的反向资金必须另建付款单或收款单。
9. 付款冲销行的 `expenseAmount` 为负数、`incomeAmount` 为空；收款冲销行的 `incomeAmount` 为负数、`expenseAmount` 为空。

## 12. 推荐后端设计

新增单一职责只读查询，例如：

```text
SupplierPurchaseFinanceFlowQueryService
SupplierPurchaseFinanceFlowQueryRepository
SupplierPurchaseFinanceFlowLineResponse
SupplierPurchaseFinanceFlowSummaryResponse
```

查询根范围固定为结算主体与供应商：

响应同时返回：

```text

```

`items` 使用 `UNION ALL` 读模型组合业务明细表和资金单据表头。第一阶段不新增泛化、可编辑的统一账本表，也不使用事件溯源；

查询必须：

1. 使用稳定 ID 关联，不用物料名称、供应商名称或单号猜测来源；
2. 由数据库完成分页前排序与 `row_number()`，避免前端分页后重新编号；
3. 按数据权限、结算主体和供应商隔离；
4. 支持按单据类型、状态、日期、物料关键字筛选；
5. 导出与页面复用同一查询口径。
6. 采购订单 ID 只能过滤业务单据链，不得用于推断或裁剪付款、收款资金行；需要查看完整资金账时必须保留根范围内全部资金单据。

默认排序固定为业务日期、审核或生效时间、稳定 ID、行号升序。筛选或分页不能改变同一结果集内的相对顺序，页面与导出必须生成一致的流转序号。

## 13. 推荐前端设计

1. 在供应商财务账簿提供统一单据流入口，先选择结算主体和供应商；采购订单详情可以携带这两个维度跳转，但不能建立订单与资金行的归属关系；

7. 导出列和页面列保持一致，并保留稳定 ID 作为隐藏审计字段；
8. 长物料名称、来源单号和金额不得挤压或遮挡其他列。

```sql
       OR status NOT IN ('草稿', '已收款')

    UNION ALL

    SELECT '供应商退款到账与通用收款单 ID 或单号冲突', COUNT(*)
    FROM public.fm_supplier_refund_receipt legacy
    JOIN public.fm_receipt receipt
      ON receipt.id = legacy.id
      OR receipt.receipt_no = legacy.refund_receipt_no

    UNION ALL

    SELECT '付款单金额或状态不兼容统一资金模型', COUNT(*)
    FROM public.fm_payment
    WHERE amount IS NULL
       OR amount <= 0
       OR status NOT IN ('草稿', '已付款')
) audit
WHERE issue_count > 0;

COMMIT;
```

V64 在 V58 迁移后逐字段复核原供应商退款到账单与通用收款单的身份、日期、方式、状态、金额、经办人、备注、版本及审计字段，
并为所有付款用途统一增加 `amount > 0` 数据库约束。V58 之前的冲突仍只能由上述只读预检提前发现，因为后续迁移无法越过一个已失败的 V58。

#### 既有资金数据的 V65 兼容桥

V58 已锁定校验和，不能调整其 SQL 顺序。V1 基线中的付款、收款状态约束只接受“已付款、已收款”，而 V58 会先把历史状态归一为
“已审核”，再替换约束。因此，存在有效历史资金单据且尚未执行 V58 的数据库，必须先通过 Flyway 单独执行
`V65__prepare_legacy_cash_status_constraints_for_v58.sql`，临时允许新旧状态并存；随后以 `outOfOrder=true` 执行 V58-V64。

V58 会把临时约束收紧为仅允许“草稿、已审核”。空库按 V1-V65 正常顺序迁移时，V65 检测到 V58 已成功执行后直接跳过，
不会改变最终 schema。禁止绕过 Flyway 手工删除状态约束，也禁止为此修改或 repair V58 校验和。

## 15. 验收标准

1. 采购订单、入库、对账、退款和收票必须按真实明细逐行展示；不论上游有多少行，每张付款单或收款单只能展示一条总额资金行；
2. 业务行通过 `documentItemId` 和 `sourceDocumentItemId` 追溯，资金行通过付款单或收款单 `documentId` 追溯；
3. 采购订单、入库、退款行的支出和收入均为空；
4. 已审核付款行只填写支出，已审核供应商收款行只填写收入；
5. 支出合计、收入合计和净现金支出与资金单据一致；
6. 计划、入库结算、对账确认、退款确认和收票金额分别汇总，不存在全部行金额合计；
7. 付款和收款不产生业务来源分配行，资金行金额必须与对应资金单据有效表头总额一致；
8. 草稿、反审核和软删除数据不进入有效汇总；
9. 页面与导出使用同一来源和汇总口径；
10. 历史供应商退款到账数据按原结算主体、供应商和金额迁移，不生成猜测式采购来源关系；
11. 同一结算主体与供应商的 `netPosition` 为零时计算为已结清，任一单据不得单独存储或覆盖该结果。
12. 付款超过当前应付时，页面必须显示正确的预付款余额；该余额只能由有效业务单据和资金单据重新计算，不能人工覆盖。
13. 新增采购应付必须按供应商总账自动消耗预付款，系统不得生成抵扣单或要求用户选择被抵扣的采购明细。
14. 预付款退款收款不得超过审核时的预付款余额；超额实际到账必须使用独立的供应商其他收款单记录。
15. 已审核资金单据不可变；冲销必须关联原资金单，累计冲销不得超额，原单与冲销单必须同时保留在单据流中。
16. 付款冲销只在支出列显示负数，收款冲销只在收入列显示负数；支出、收入和净现金支出汇总必须包含冲销影响。
17. 未完成采购订单的入库明细不得出现在供应商对账候选中；对账审核不得重复增加应付或产生现金收支。
18. 同一供应商对账单可以覆盖多张完成采购订单，但任一明细不得跨供应商、跨结算主体或丢失原采购入库行来源。
19. 对账草稿预占、修改释放、删除释放和审核转正式必须金额守恒；并发草稿不得使任一入库明细的累计对账金额超过其结算金额。
20. 新建对账单时，同一采购入库明细必须一次对账全部剩余金额，前后端均不得提供减少本次对账金额或拆分对账的入口。

## 17. 采购、销售、物流主线问答

### 17.1 已确认决策

1. 销售订单同时支持正常销售和预售两种业务模式，不能把两者视为同一状态含义；
2. 正常销售只能引用“完成采购”订单已经实际入库的货源，不得把未入库或已转采购退款的剩余数量带入销售；
3. 预售允许引用“已审核但尚未完成采购”的采购订单货源，销售订单必须明确展示预售标识；
4. 预售可以先创建销售订单，但来源货物尚未完成实际采购入库时，销售出库只能保持“草稿”，不得审核生效；新流程禁止再写入历史“预出库”状态；
5. 后端必须根据销售订单表头的正常销售或预售模式分别校验来源：正常销售引用已完成采购的有效入库明细，预售引用已审核且尚未完成采购的采购订单明细；
6. 销售订单必须在表头明确选择“正常销售”或“预售”，同一张销售订单内不允许混合两种模式的明细；
7. 预售订单后续货源全部满足时，只解除销售出库限制，不改写原销售模式或来源标识，保证历史上仍可识别该订单由预售产生；
8. 预售数量超过来源采购订单最终实际入库数量时，系统必须阻止该采购订单完成采购，不能自动减少销售订单或允许来源容量变为负数；
9. 用户可以在完成采购前直接修改预售销售订单数量以消除货源缺口，保存时后端必须重新锁定并校验采购来源容量；
10. 销售订单数量只能在销售订单处于“草稿”时直接修改；已审核销售订单必须先满足下游保护并反审核为草稿，不能在已审核状态直接修改数量；
11. 存在未删除的关联销售出库单时，销售订单不得反审核或修改数量；销售出库已审核时，必须先按合法流程处理销售出库，不能绕过下游保护；
12. 关联销售出库全部进入删除状态后，不再占用销售订单数量，并由删除销售出库命令联动把销售订单自动退回“草稿”；
13. 关联销售出库处于“已审核”时，销售订单数量及其他来源行字段锁定，仅允许调整送货日期、备注和单价；
14. 销售出库从草稿审核后，来源销售订单自动从“已审核”进入“交付核定”；
15. 销售出库从已审核反审核为草稿后，来源销售订单自动从“交付核定”回退为“已审核”；
16. 删除销售出库后，若来源销售订单已不存在其他未删除销售出库，销售订单自动从“已审核”回退为“草稿”；删除下游与更新上游必须处于同一事务，任一步失败则整体回滚；
17. 未删除的销售订单草稿即预占其采购来源数量和重量；修改时排除当前订单自身后重新计算容量，删除后释放预占，禁止两个并发草稿超占同一来源；
18. 当前流程不允许同一张销售订单分批生成多张物流单或销售出库。物流单必须完整导入一张已审核销售订单的全部明细和数量，漏行、重复行、混入其他订单或篡改来源快照均拒绝；
19. 销售出库只能由未审核物流单通过“生成销售出库”命令产生草稿，公共创建接口不得伪造物流来源；同一物流单只能存在一张活动销售出库；
20. 物流生成的销售出库锁定来源、物料、仓库、数量和单价来源，只允许在草稿中调整出库日期、备注和实际出库重量，金额按实际重量与来源单价重新计算；
21. 销售出库审核前必须再次校验来源采购已经完成、销售订单仍为已审核、来源未被占用，并且出库完整覆盖整张销售订单；审核成功后销售订单进入“交付核定”；
22. 物流单审核必须存在已审核销售出库，并以销售出库实际重量同步物流重量和运费；生成销售出库后物流单的来源和明细受下游保护；
23. 销售出库反审核为草稿时，销售订单从“交付核定”回退“已审核”；删除草稿销售出库时，同时软删除其来源物流单，并在不存在其他活动出库时把销售订单回退“草稿”；物流已审核或存在对账等下游时禁止删除；
24. “完成销售”只能通过专用命令从“交付核定”进入“完成销售”，完成销售为终态，不允许普通保存、通用状态接口或下游联动静默改写；
25. 前端按钮必须以权限、当前状态和后端返回的稳定关联 ID共同判断：完成采购只在已审核采购订单显示，撤销完成只在完成采购显示，生成销售出库只在未审核且尚无关联出库的物流单显示；后端始终重复执行同一业务校验，不能信任按钮隐藏；
26. V61 将历史“预出库”等遗留销售订单状态统一归一为“交付核定”；迁移后状态常量只保留只读兼容，任何新保存路径都不得再产生“预出库”。
