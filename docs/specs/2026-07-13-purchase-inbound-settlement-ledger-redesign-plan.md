# 采购入库拆分与采购结算记账重构计划

## 1. 文档状态

- 状态：已部分废止，仅保留与新采购财务决策不冲突的内容作为参考
- 日期：2026-07-13
- 涉及仓库：后端 `leo`、前端 `aries`
- 发布方式：新旧模型一次性整体切换
- 财务边界：继续使用业务应收应付子账，不引入复式记账、总账凭证和会计期间
- 数据迁移：禁止按本计划删除采购退款单；供应商退款到账旧模块只能在数据迁入通用收款单并验证后移除

### 1.1 决策替代声明（2026-07-14）

本计划早期提出的“删除采购退款单并改为不可编辑记账事件”已经被
[采购财务逐行单据流明细改造计划](2026-07-13-purchase-finance-document-flow-detail-plan.md)明确否决。
发生冲突时，必须以后者为准，禁止继续实施以下旧决策：

1. 删除采购退款单业务模块、表、接口、权限或编号规则；
2. 用泛化记账事件替代采购退款单的“草稿 -> 已审核”业务流程；
3. 把供应商退款到账迁移为缺少原资金单据的匿名记账事件；
4. 在历史数据尚未完整迁入通用收款单前删除供应商退款到账旧模块。

采购入库按过磅规则拆分、部分入库、完成采购入口和超差校验等不冲突内容仍可参考。供应商退款到账独立模块最终可以移除，
但前提是历史数据已经迁入通用收款单并通过金额守恒验证；这不等于允许删除采购退款单。

## 2. 背景与问题

当前采购入库和采购退款链路存在以下问题：

1. 采购订单同时包含盘螺、线材、螺纹钢或其他直条商品时，可以被导入同一张采购入库单。头部或行级结算方式传播会导致本不需要过磅的直条商品也要求填写过磅重量。
2. 盘螺等过磅商品产生的上差、下差已经能够在采购入库中计算，但后续又通过人工采购退款单表达未入库或结算差额，形成重复业务概念和额外操作。
3. 采购退款单继续被供应商退款到账、预付款容量、入库审核限制、收票容量、权限、编号规则和主数据引用保护依赖，不能只隐藏前端菜单。
4. 当前应收应付主要按业务单据汇总，采购入库在财务明细中容易只显示单据级记录，无法清晰解释每个商品的理论金额、过磅差额和最终结算金额。
5. 现有应收应付是业务子账读模型，不是复式记账。若同时按实际重量确认较低应付并额外登记同一笔退款，会造成重复冲减。
6. 草稿转审核、反审核、订单结单和取消结单需要形成可审计的自动记账与反向冲销，不能通过覆盖或删除历史记录完成。

本次重构以采购订单为原始约定，以采购入库为实际履约，以不可编辑的商品级业务记账事件表达基础应付、过磅差额、未入库退款、付款、预存余额和冲销。

## 3. 目标

### 3.1 业务目标

1. 导入采购订单时由后端按过磅规则和仓库自动拆分采购入库草稿。
2. 盘螺、线材等需要过磅的商品只进入过磅入库单；螺纹钢、圆钢直条等理算商品只进入理算入库单。
3. 用户不再手工制作采购退款单。过磅补退款在入库审核时自动记账，未入库退款在采购订单结单时自动记账。
4. 退款先冲减同一供应商、同一结算主体下的未付应付；超出部分默认形成供应商预存余额，也允许在结单时选择转为供应商应收。
5. 财务汇总按供应商和结算主体展示，财务明细按商品行和记账事件平铺展示。
6. 自动记录不可编辑、不可删除；撤销业务时生成反向冲销记录。
7. 删除采购退款业务模块和供应商退款到账独立模块，并将历史记录迁移为只读记账事件。

### 3.2 工程目标

1. 过磅规则、导入拆分、剩余数量校验、最后一张入库判断和结单计算全部由后端权威执行。
2. 同一采购订单的导入批次原子创建，任一草稿创建失败时整体回滚。
3. 所有正式财务事件具有稳定来源 ID、商品快照、结算主体和冲销关系。
4. 历史迁移前后满足供应商级、结算主体级、采购订单级和商品级金额守恒。
5. 新增 Flyway 完成迁移，不修改已经执行的 `V1` 或任何 post-baseline 脚本。

## 4. 非目标

1. 不建设会计科目、借贷分录、总账凭证、试算平衡、会计期间或财务报表。
2. 不提供预存余额到具体采购订单的逐笔分配或先进先出核销。
3. 不允许按采购订单商品行分别结单；结单始终作用于整张采购订单。
4. 不保留可新增、可编辑的采购退款单或供应商退款到账单。
5. 不允许前端自行判断品类是否过磅、是否为最后一张入库单或最终补退款金额。
6. 不在本次重构中改变销售、物流等其他应收应付来源的业务语义，除非它们必须适配新的统一记账事件读模型。

## 5. 已确认决策

| 主题 | 决策 |
|---|---|
| 原始业务单据 | 采购订单保持原始约定，不因入库或补退款改写原始数量、重量、单价和金额 |
| 入库拆分 | 后端按过磅规则、仓库、供应商和结算主体分组 |
| 同规则品类 | 同仓库下，盘螺和线材可以合并为过磅单；螺纹钢和圆钢直条可以合并为理算单 |
| 批次创建 | 用户确认预览后，后端在单个事务中生成全部拆分草稿 |
| 审核方式 | 拆分后的每张入库单独立审核，不提供一键批量审核 |
| 默认数量 | 草稿默认带入全部剩余数量，用户可以减少 |
| 移除商品 | 允许从当前草稿移除商品，数量返回采购订单待入库 |
| 空草稿 | 保存时自动取消空草稿并释放占用，不保留可编辑空单 |
| 最后一张判断 | 按整个采购订单判断，不按当前导入批次或仓库判断 |
| 结单入口 | 只在整个采购订单不存在其他活动入库草稿时，于最后一张入库单审核区显示 |
| 结单默认值 | 剩余数量为零时自动完成；剩余数量大于零时默认不勾选结单 |
| 结单权限 | 具有采购入库审核权限的用户均可审核并结单 |
| 结单范围 | 整张采购订单一起结单，所有剩余商品同时终止入库 |
| 取消结单 | 无不可逆下游依赖时允许取消，并生成反向冲销记录 |
| 超差处理 | 超过品类上下差容限时允许审核，但必须二次确认并留痕 |
| 超差原因 | 必选标准原因，可填写补充备注；选择“其他”时备注必填 |
| 财务模型 | 保留应收应付业务子账，使用不可变业务记账事件，不引入复式记账 |
| 退款处理 | 先冲减未付应付，超出部分形成供应商预存余额或供应商应收 |
| 默认余额处理 | 默认形成供应商预存余额 |
| 预存余额使用 | 只在供应商和结算主体汇总层面计算净额，不分配到具体采购订单 |
| 财务汇总 | 应收应付列表继续按往来单位和结算主体汇总 |
| 财务明细 | 每个商品、每种记账事件分别平铺为只读行 |
| 盘螺记账 | 基础入库应付和过磅上差/下差拆成两条独立事件 |
| 撤销方式 | 保留原事件，新增反向冲销事件，不覆盖、不删除原事件 |
| 历史迁移 | 全量迁移全部历史有效业务数据 |
| 历史采购退款 | 转换为只读历史记账事件，保留原退款单号和 ID，随后删除模块和表 |
| 历史退款到账 | 转换为财务结算事件，保留原到账单号和 ID，随后删除独立页面和表 |
| 发布方式 | 后端、前端和数据库一次性整体切换，不并行运行新旧模型 |

## 6. 领域模型

### 6.1 采购订单

采购订单继续表达供应商原始承诺：

- 原始数量；
- 原始理论重量；
- 原始单价和金额；
- 供应商和结算主体；
- 商品与仓库约定。

新增结单审计字段，字段名以实施时现有命名规范为准：

```text
closed_flag
closed_at
closed_by
closed_name
close_reason
close_source_inbound_id
```

约束：

1. 结单不修改采购订单明细原始值。
2. 结单后禁止创建新采购入库批次。
3. 取消结单必须先模拟全部冲销后的供应商余额，存在不可逆依赖时拒绝。
4. 结单和取消结单都必须锁定采购订单及相关明细，避免并发导入或审核。

### 6.2 入库导入批次

新增轻量导入批次，用于关联一次导入产生的多张草稿，不承担财务含义：

```text
purchase_inbound_import_batch
- id
- source_purchase_order_id
- batch_no
- created_by / created_name / created_at
```

采购入库增加可空 `import_batch_id`。历史入库不强制回填批次。

批次仅用于：

- 原子创建结果回显；
- 前端展示同批进度；
- 排查一次导入产生的单据；
- 防止前端把拆分结果误认为一张单据。

“最后一张入库单”仍按整个采购订单判断，不能仅根据批次判断。

### 6.3 商品级业务记账事件

建议新增 `fm_counterparty_ledger_event`，作为不可编辑的业务子账事件表。最终表名必须与现有财务表前缀和命名规范保持一致。

建议字段：

```text
id
event_type
event_effect
ledger_bucket
amount
business_date

counterparty_type
counterparty_id
counterparty_code
counterparty_name
settlement_company_id
settlement_company_name

source_module
source_document_id
source_document_no
source_item_id
source_line_no

material_id
material_code
brand
category
material
spec
length
unit
quantity
quantity_unit
batch_no
warehouse_id
warehouse_name

theoretical_weight_ton
actual_weight_ton
weight_adjustment_ton
unit_price

reversal_of_event_id
generation_mode
calculation_snapshot_json

created_by
created_name

```

核心约束：

1. `amount > 0`，方向由 `event_effect` 表达，禁止同时使用负金额和方向字段造成双重符号。
2. `generation_mode` 至少支持 `SYSTEM_AUTO`、`SYSTEM_REVERSAL`、`HISTORY_MIGRATION`、`MANUAL_ADJUSTMENT`。
3. 冲销事件必须填写 `reversal_of_event_id`，原事件不可修改。
4. 自动事件使用稳定幂等键，建议由 `source_module + source_item_id + event_type + source_revision` 组成唯一约束。
5. 所有商品快照从来源单据取得，历史显示不跟随主数据改名。
6. 自动事件只由领域服务写入，不开放通用 CRUD 保存接口。
7. 人工台账调整继续通过现有调整单审核产生事件，不允许直接编辑事件表。

建议事件类型：

```text
PURCHASE_INBOUND_BASE
PURCHASE_WEIGH_OVER_ADJUSTMENT
PURCHASE_WEIGH_UNDER_ADJUSTMENT
PURCHASE_ORDER_CLOSE_REFUND
PURCHASE_PAYMENT_SETTLEMENT
PURCHASE_PREPAYMENT_SETTLEMENT
SUPPLIER_RECEIVABLE_TRANSFER
HISTORICAL_PURCHASE_REFUND
HISTORICAL_SUPPLIER_REFUND_RECEIPT
MANUAL_LEDGER_ADJUSTMENT
REVERSAL
```

建议账簿桶：

```text
PAYABLE
RECEIVABLE
SUPPLIER_CREDIT
```

供应商预存余额不做采购订单级分配。读模型按 `supplier_id + settlement_company_id` 汇总应付、应收和预存余额，再计算净头寸。

## 7. 金额口径与不变量

### 7.1 入库商品计算

理算商品：

```text
theoreticalWeight = quantity * pieceWeightTon
baseAmount = theoreticalWeight * unitPrice
actualWeight = null
weightAdjustment = 0
adjustmentAmount = 0
settlementAmount = baseAmount
```

过磅商品：

```text
theoreticalWeight = quantity * pieceWeightTon
actualWeight = weighWeightTon
weightAdjustment = actualWeight - theoreticalWeight
baseAmount = theoreticalWeight * unitPrice
adjustmentAmount = weightAdjustment * unitPrice
settlementAmount = baseAmount + adjustmentAmount
```

记账事件：

- `PURCHASE_INBOUND_BASE`：基础应付增加 `baseAmount`；
- 上差：`PURCHASE_WEIGH_OVER_ADJUSTMENT`，应付增加 `abs(adjustmentAmount)`；
- 下差：`PURCHASE_WEIGH_UNDER_ADJUSTMENT`，应付减少 `abs(adjustmentAmount)`；
- 理算商品不生成过磅调整事件。

严禁同时按 `settlementAmount` 生成基础应付，又额外生成过磅调整，否则会重复记账。

### 7.2 订单结单计算

每个采购订单商品行分别计算：

```text
remainingQuantity = max(orderedQuantity - effectiveInboundQuantity, 0)
orderedTheoreticalAmount = orderedQuantity * pieceWeightTon * unitPrice
effectiveInboundBaseAmount = sum(baseAmount)
effectiveWeightAdjustmentAmount = sum(adjustmentAmount)
effectiveInboundSettlementAmount = effectiveInboundBaseAmount + effectiveWeightAdjustmentAmount
remainingRefundAmount = remainingQuantity * pieceWeightTon * unitPrice
```

结单退款只处理未入库数量对应的理论金额。已入库商品的过磅上差或下差已在入库审核时通过独立调整事件记账，不得再次并入结单退款，否则会重复补款或重复退款。

旧 `PurchaseRefundCalculator` 当前按“订单理论金额减入库实际结算金额”计算退款，包含已入库商品的磅差。实施新模型时必须替换该口径，不能直接复用旧计算器。

只有状态为已审核或完成入库且未删除的采购入库参与正式计算。草稿只参与预览，不参与正式账簿。

结单为整单操作，但生成的 `PURCHASE_ORDER_CLOSE_REFUND` 必须按采购订单商品行分别落事件。

### 7.3 供应商净头寸

同一供应商和同一结算主体内：

```text
grossPayable = payableIncrease - payableDecrease - settledPayment
grossReceivable = receivableIncrease - receivableDecrease - receivedSettlement
supplierCredit = creditIncrease - creditDecrease
```

默认结单方式为供应商预存余额：

1. 退款事件先减少应付；
2. 汇总后应付头寸低于零的部分展示为供应商预存余额；
3. 后续新应付在供应商汇总层面自然抵扣预存余额；
4. 不生成预存余额到采购订单的分配明细。

若用户在结单时选择“转供应商应收”，系统在结单事务内根据当前供应商头寸拆分应付冲减和供应商应收事件。拆分结果写入计算快照，后续撤销时逐条反向冲销。

### 7.4 守恒公式

商品行：

```text
settlementAmount = baseAmount + adjustmentAmount
```

采购订单：

```text
originalOrderAmount
= effectiveInboundBaseAmount
  + closeRefundAmount

effectiveInboundSettlementAmount
= effectiveInboundBaseAmount
  + effectiveWeightAdjustmentAmount
```

允许因金额精度产生不超过最小货币单位的舍入差异，差异处理必须使用现有统一金额精度，不新增临时四舍五入规则。

供应商和结算主体：

```text
openingPosition
+ recognitionEvents
- settlementEvents
+ adjustmentEvents
+ reversalEvents
= closingPosition
```

## 8. 采购入库拆分流程

### 8.1 权威分组键

```text
purchaseOrderId
+ supplierId
+ settlementCompanyId
+ warehouseId
+ purchaseWeighRequired
```

其中 `purchaseWeighRequired` 从后端物料品类规则读取。不得硬编码“盘螺”“线材”“螺纹钢”等名称进行分组。

同一分组中的不同品类可以合并为一张入库单。不同仓库或不同过磅规则必须拆单。

### 8.2 导入预览

新增后端预览命令，输入采购订单 ID，返回：

- 采购订单基本信息；
- 当前剩余数量；
- 每个拆分组；
- 每组商品明细；
- 结算方式；
- 仓库；
- 预计生成草稿数量；
- 阻止导入的原因。

预览只读，不占用数量。确认创建时必须重新锁定并计算，不能直接信任预览结果。

### 8.3 原子创建

用户确认后调用批次创建命令：

1. 锁定采购订单和所有来源明细；
2. 校验采购订单未结单；
3. 重新汇总其他活动入库占用；
4. 按权威分组键拆分；
5. 创建导入批次；
6. 为每个非空分组创建采购入库草稿；
7. 返回批次和草稿列表；
8. 任一步失败则整体回滚。

### 8.4 草稿编辑

1. 数量默认使用全部剩余数量。
2. 用户只能减少数量，不能超过来源剩余数量。
3. 过磅重量必须对应本次实际入库数量。
4. 用户可以从本次草稿移除商品，数量释放回待入库。
5. 不允许在理算单中添加过磅商品，也不允许在过磅单中添加理算商品。
6. 明细全部移除后，保存动作将草稿取消并释放占用。
7. 已分配正式编号的空草稿保留编号审计并标记已取消；未形成正式编号时不得重复消费编号。

## 9. 审核、结单与超差

### 9.1 审核命令

采购入库审核已包含结单和超差确认参数，不应继续仅使用通用字符串状态更新接口。建议新增专用审核命令：

```json
{
  "closePurchaseOrder": false,
  "remainingBalanceDisposition": "SUPPLIER_CREDIT",
  "overToleranceConfirmations": [
    {
      "inboundItemId": 1,
      "reasonCode": "TRANSPORT_LOSS",
      "remark": ""
    }
  ]
}
```

审核事务必须：

1. 锁定采购订单及来源明细；
2. 重新验证剩余数量；
3. 重新读取品类过磅规则和容差；
4. 后端重算理论重量、实际重量、磅差和金额；
5. 校验当前单据是否可以作为最后一张结单单据；
6. 保存超差确认快照；
7. 将入库状态改为已审核或完成入库；
8. 生成商品级基础应付和过磅调整事件；
9. 若结单，生成所有剩余商品的未入库退款事件；
10. 同步采购订单实际重量和逐件重量账本；
11. 保证数据库状态、逐件重量和记账事件在同一事务内一致。

### 9.2 最后一张入库判断

只有同时满足以下条件才显示并接受结单：

1. 当前入库单为可审核草稿；
2. 当前采购订单不存在其他活动入库草稿；
3. 不存在其他正在处理的导入批次；
4. 当前用户具有采购入库审核权限；
5. 当前审核不会违反退款、付款、销售锁定或其他容量约束。

前端可以提前隐藏或禁用，但后端必须在提交时重新判断。

### 9.3 结单交互

- 剩余数量为零：不要求用户额外勾选，审核按钮显示“审核并完成订单”。
- 剩余数量大于零：默认不勾选“本次审核后终止该采购订单的后续入库”。
- 用户勾选后展示逐商品未入库差额、过磅差额、应付冲减、供应商预存余额或供应商应收预览。
- 主按钮变为“审核并结单”，提交前二次确认。

### 9.4 超差确认

标准原因代码至少包含：

```text
TRANSPORT_LOSS
HANDLING_LOSS
MEASUREMENT_DIFFERENCE
SUPPLIER_CONFIRMED
MOISTURE_OR_IMPURITY_CHANGE
THEORETICAL_WEIGHT_DEVIATION
OTHER
```

每条超差商品分别保存：

- 理论重量；
- 实际重量；
- 上差/下差；
- 容差配置快照；
- 实际差异百分比；
- 原因代码和备注；
- 确认人和确认时间。

选择 `OTHER` 时备注必填。前端可以提供“应用到全部超差行”，后端仍需按商品行保存。

### 9.5 反审核和取消结单

1. 原始自动记账事件不删除、不修改。
2. 为每条原事件生成一条反向冲销事件。
3. 反审核后同步撤销实际重量对逐件重量账本的影响。
4. 取消结单后采购订单恢复可入库，未入库退款逐条冲销。
5. 若冲销后供应商头寸不足、存在不可逆外部结算或销售锁定冲突，后端拒绝操作并返回具体差额。
6. 再次审核或结单时重新计算并生成新事件，不复用旧事件。

完成态不能成为不可撤销的死状态，状态转换明确为：

```text
采购入库：完成入库 -> 草稿
来源采购订单：完成采购 -> 已审核
采购订单继续反审核：已审核 -> 草稿
```

采购入库从“完成入库”反审核时，必须在同一事务内撤销入库影响并重新计算来源采购订单。采购订单从“完成采购”执行反审核时，第一次操作定义为“重新打开”并恢复到“已审核”；用户确认还要修改原始订单后，再执行一次反审核进入“草稿”。不得直接把仍有有效采购入库的完成采购订单改成草稿。

前端批量反审核必须根据模块和当前状态逐条解析目标状态，不能只允许当前状态等于“已审核”的记录，也不能对一批不同完成状态统一发送同一个目标状态。

## 13. Flyway 与历史数据迁移

### 13.1 迁移原则

1. 当前最大迁移版本在实施前重新确认；按本计划编写时为 `V56`，下一版本应从 `V57` 开始。
2. 禁止修改 `V1__baseline.sql` 和任何已执行的 `V2` 至 `V56`。
3. 禁止恢复旧迁移链。
4. 所有持久化 DDL/DML 必须通过 Flyway。
5. 一次性发布可以包含多个连续 Flyway 脚本，但应用只能在全部迁移成功后启动。

### 13.2 建议迁移序列

实际版本号以开发开始时最大版本为准：

```text
V57__add_purchase_inbound_import_batch_and_order_close.sql
V58__add_counterparty_ledger_event.sql
V59__backfill_counterparty_ledger_events.sql
V60__migrate_purchase_refunds_and_refund_receipts.sql
V61__remove_purchase_refund_and_refund_receipt_modules.sql
```

职责：

- `V57`：新增导入批次、采购订单结单字段、必要约束和索引；
- `V58`：新增不可变记账事件表、枚举约束、冲销外键、幂等唯一约束和查询索引；
- `V59`：从采购入库、付款、预付款和人工台账调整回填商品级事件；
- `V60`：迁移历史采购退款和供应商退款到账为历史事件；
- `V61`：验证迁移计数和余额后，清理旧菜单、权限、编号规则、外键、索引和业务表。

### 13.3 历史状态映射

采购入库：

- 已审核、完成入库：生成基础应付和过磅调整事件；
- 草稿：不生成正式事件；
- 已删除：不生成有效事件。

采购退款：

- 已审核且未删除：生成 `HISTORICAL_PURCHASE_REFUND`；
- 草稿：仅保留迁移快照，不影响余额；
- 已删除：不影响余额。

供应商退款到账：

- 已收款且未删除：生成 `HISTORICAL_SUPPLIER_REFUND_RECEIPT`；
- 草稿或已删除：不影响余额。

历史迁移事件保留原模块、原 ID、原单号、原业务日期和计算来源，不允许通过新接口撤销。

### 13.4 迁移验证门禁

在删除旧表前，迁移 SQL 必须主动失败并回滚以下异常：

1. 有效旧单据无法匹配稳定供应商 ID；
2. 有效旧单据无法匹配结算主体；
3. 采购退款明细无法匹配采购订单商品；
4. 迁移事件数量与来源有效明细数量不一致；
5. 供应商和结算主体迁移前后净余额不一致；
6. 采购订单原始理论金额、有效入库基础金额和结单退款不守恒，或入库基础金额、磅差调整与实际结算不守恒；
7. 同一来源生成重复幂等事件；
8. 历史到账累计超过可解释的退款或预存余额且没有明确迁移策略。

迁移前必须执行只读审计 SQL，并归档异常数据清单。不得为了让迁移通过而在生产库直接手工修改数据。

## 14. 查询与性能

建议索引：

```text
(counterparty_type, counterparty_id, settlement_company_id, business_date, id)
(source_module, source_document_id)
(source_module, source_item_id)
(event_type, business_date)
(material_id, business_date)
(reversal_of_event_id)
```

要求：

5. 历史迁移采用集合 SQL，避免逐行应用层回填。

### 15.4 端到端验收场景

场景一：混合采购订单。

1. 订单包含盘螺、线材、螺纹钢和圆钢，分布在两个仓库；
2. 导入预览按规则和仓库拆分；
3. 后端一次生成全部草稿；
4. 理算单无过磅输入，过磅单必须填写实磅；
5. 分别审核后财务按商品生成基础和差额事件。

场景二：部分入库后结单。

1. 默认带入全部剩余数量；
2. 用户减少本次数量并移除一个商品；
3. 其他草稿处理完毕后，最后一张显示结单选项；
4. 用户勾选结单；
5. 系统按所有剩余商品生成未入库退款；
6. 退款先冲减应付，剩余形成供应商预存余额。

场景三：超差审核。

1. 实磅超过品类容差；
2. 前端展示实际比例和预计补退款；
3. 用户选择标准原因并确认；
4. 后端保存快照并按真实重量审核；
5. 财务商品行显示超差标识。

场景四：反审核和取消结单。

1. 原事件保持不变；
2. 系统逐条生成反向冲销；
3. 采购订单恢复可入库；
4. 财务余额恢复且审计链完整。

场景五：历史迁移。

1. 有效历史入库生成商品级事件；

### 16.3 回滚

由于本次迁移会删除旧业务表，应用回滚必须与数据库备份恢复绑定，禁止只回滚应用版本。

回滚触发条件：

- Flyway 失败；
- 余额守恒失败；
- 有效历史来源无法追溯；
- 新后端无法完成关键冒烟；
- 财务明细与汇总不一致。

回滚步骤：

1. 保持业务冻结；
2. 停止新应用；
3. 恢复发布前数据库备份；
4. 部署旧后端和旧前端；
5. 校验旧系统余额快照；
6. 解除冻结并记录失败原因。

## 17. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| 一次性切换范围大 | 应用与数据库无法独立回滚 | 维护窗口、生产副本演练、数据库与应用绑定回滚 |
| 历史余额口径不一致 | 迁移无法守恒 | 发布前只读审计、异常清单、迁移 SQL 主动失败 |
| 重复记账 | 应付被重复增加或冲减 | 基础金额与调整金额分事件、稳定幂等键、商品级守恒测试 |
| 并发重复入库 | 超出采购订单数量 | 采购订单明细锁、创建和审核时重新计算 |
| 前端规则缓存过期 | 错误拆分或错误输入 | 拆分、容差和最后一张判断全部以后端为准 |
| 冲销后余额不足 | 供应商头寸失真 | 操作前模拟余额、存在不可逆依赖时拒绝 |
| 商品明细表数据量增加 | 查询和导出变慢 | 事件快照、数据库分页、复合索引、执行计划验证 |
| 旧单号不可查询 | 历史审计中断 | 历史事件保留原模块、原 ID 和原单号 |

## 20. 跨模块状态机与下游保护整改

本章节记录 2026-07-13 对采购、销售、合同、物流和财务模块状态链的补充审查结果。该整改是采购入库与记账重构的前置门禁：来源单据状态和下游占用关系不可靠时，不允许进入自动记账与历史迁移实施阶段。

### 20.1 当前缺口

| 模块 | 当前缺口 | 风险 |
|---|---|---|
| 采购订单 | 反审核和删除只检查采购退款、预付款与收票，未检查活动采购入库和来源销售订单 | 下游单据继续引用草稿或已删除的采购订单 |
| 采购入库 | 完成入库反审核和删除未检查供应商对账单占用 | 对账单保留失效的来源入库明细 |
| 销售订单 | 反审核和删除只检查开票，未检查活动销售出库 | 销售出库继续引用草稿或已删除的销售订单 |
| 采购合同、销售合同 | 后端声明已签署到执行中、已归档的迁移，但没有状态接口，且已签署状态禁止普通编辑 | 状态迁移不可达，合同签署后无法反签或归档 |
| 采购合同导入 | 从采购订单导入时默认写入已归档 | 创建接口拒绝直接写入终态，导入后无法保存 |
| 前端通用状态保护 | 缺少已开票、已收票和已归档 | 前端显示无效的编辑、删除入口，提交后才由后端拒绝 |

### 20.2 统一规则

1. 来源单据从有效态退出或删除前，必须检查所有未删除下游单据，不以其草稿或正式状态区分占用；下游单据存在即拒绝，错误信息必须指出具体下游模块和处理方式。
2. 下游占用检查、来源行锁定和状态更新必须在同一事务内执行，避免检查后并发创建下游单据。
3. 状态迁移由后端模块专属状态机权威定义；前端只根据后端能力和当前状态展示操作，不允许通过普通编辑器直接写入完成态或终态。
4. 反审核只撤销当前业务阶段，不隐式删除下游单据。需要撤销时，用户必须按业务链从最下游向上游处理。
5. 删除校验至少与反审核校验等强；反审核不允许的下游占用，删除同样不允许。
6. 前后端受保护状态必须共享同一语义清单，并由契约测试验证；后端仍作为最终安全边界。

### 20.3 后端实施

1. 新增采购订单下游变更保护，统一检查活动采购入库和来源销售订单；在反审核为草稿、删除和会改变来源行的编辑操作中复用。
2. 新增采购入库下游变更保护，使用供应商对账单来源入库占用查询；覆盖完成入库反审核、草稿删除和来源明细变更。
3. 新增销售订单下游变更保护，使用销售出库来源订单行查询；覆盖反审核为草稿、删除和来源明细变更。
4. 保留现有采购退款、预付款、收票、开票、收付款、物流和对账保护，并将同一模块的保护收敛到单一领域服务，避免状态接口与删除接口规则漂移。
5. 为采购合同和销售合同增加受 `audit` 权限保护的专用状态接口，支持且仅支持以下迁移：

```text
草稿 -> 执行中
执行中 -> 草稿
执行中 -> 已签署
已签署 -> 执行中
已签署 -> 已归档
```

6. 合同普通更新只修改可编辑业务字段，不承担已签署、反签和归档动作；已归档保持不可逆终态。
7. 所有下游占用查询忽略已删除记录，返回稳定的存在性或来源 ID 集合，并配合来源锁避免并发穿透。

### 20.4 前端实施

1. 将已开票、已收票和已归档补入编辑、删除保护集合，并保持模块级覆盖能力。
2. 合同页面提供执行、签署、反签和归档的显式状态命令；按钮按当前状态和 `audit` 权限显示，归档动作必须二次确认。
3. 采购合同从采购订单导入后状态固定为草稿，不允许父单映射覆盖为已归档。
4. 批量审核和反审核继续逐条解析目标状态；合同签署和归档不复用语义含混的通用“审核”按钮。
5. 后端返回下游占用错误时，前端保留具体业务提示，不统一改写为“操作失败”。

### 20.5 验收场景

1. 已审核采购订单存在草稿、已审核或完成入库单时，反审核和删除均被拒绝。
2. 已审核采购订单存在来源销售订单时，反审核和删除均被拒绝；删除全部下游单据后可以正常反审核。
3. 完成入库已生成待确认或已确认供应商对账单时，反审核和删除均被拒绝；删除对账单后可以反审核。
4. 已审核销售订单存在草稿、预出库或已审核销售出库时，反审核和删除均被拒绝。
5. 采购合同和销售合同可以完整走通草稿、执行中、已签署、反签和已归档状态链，非法跨级调用返回明确错误。
6. 从采购订单导入采购合同后可以直接保存草稿，不再发送已归档状态。
7. 已开票、已收票和已归档记录在列表、行操作、批量操作和编辑器中均不显示编辑或删除入口。
8. 并发场景下，一个事务检查来源可撤销的同时，另一个事务不能成功创建新的下游占用后让来源撤销提交。
