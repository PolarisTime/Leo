# 打印流程逻辑审计与整改建议

## 1. 文档目的

本文梳理当前系统从业务单据选择、模板查询、后端输出生成到浏览器或 C-Lodop 执行打印的完整链路，并记录现状、权限边界、数据安全风险和分阶段整改建议。

审计结论：当前前端所称的“打印作业”本质上是一次**同步打印设置与输出生成流程**，不是后台打印作业系统。系统目前没有打印作业表、队列、状态机、重试、客户端完成回执或物理打印结果历史，因此只能证明系统生成过打印输出，不能证明打印机已成功出纸。

## 2. 审计范围

本次审计覆盖：

- 前端业务列表的打印入口与权限展示；
- 打印设置弹窗、模板选择、字段设置及输出执行；
- PDF 与 C-Lodop 两条输出通道；
- 后端打印模板、业务数据装配和输出生成接口；
- 模板的文件、数据库和 Flyway seed 管理；
- 操作日志、状态约束、错误处理和测试覆盖。

本次审计不包含：

- 对实体打印机、C-Lodop 服务或浏览器打印驱动的现场联调；
- 对打印模板视觉效果和纸张套打坐标的逐模板验收；
- 打印作业中心、队列或打印客户端的代码实现。

## 3. 当前端到端调用链

```text
业务列表勾选单据
  → 打开 PrintJobModal
  → GET /print-templates 查询可用模板
  → POST /print/items 查询可打印的商品明细
  → 用户确认打印设置
  → POST /print/record 生成打印输出
  → 后端读取模板、单据及明细
      ├─ PDF_FORM：iText 填充 PDF，返回 PDF Base64
      └─ COORD：返回白名单 LODOP 脚本与业务数据
  → 前端执行输出
      ├─ PDF：新窗口预览、下载或 iframe.print
      └─ COORD：解析白名单命令，调用 C-Lodop PREVIEW/PRINT
```

关键实现位置：

- 前端业务入口：`aries/src/views/modules/BusinessGridRouteContent.tsx`
- 前端打印设置：`aries/src/views/modules/components/PrintJobModal.tsx`
- 前端业务打印动作：`aries/src/hooks/useBusinessGridPrintActions.ts`
- 前端输出执行器：`aries/src/utils/print-output-runner.ts`
- C-Lodop 适配器：`aries/src/utils/clodop.ts`
- 后端打印接口：`src/main/java/com/leo/erp/system/printtemplate/web/PrintScriptController.java`
- 后端输出服务：`src/main/java/com/leo/erp/system/printtemplate/service/PrintOutputService.java`
- 后端脚本服务：`src/main/java/com/leo/erp/system/printtemplate/service/PrintScriptService.java`

### 3.1 PDF 输出

后端使用 iText 同步读取 PDF 表单模板并填充单据数据，再将完整 PDF 编码成 Base64 放入 JSON 响应。前端根据用户动作进行预览、下载或调用浏览器打印。

该方式实现简单，但 Base64 相比原始二进制通常增加约 33% 的传输体积。大模板或批量打印场景下，应优先评估直接返回 `application/pdf`、短期下载地址或流式响应，避免 JSON 和前后端内存占用持续放大。

### 3.2 C-Lodop 输出

后端返回 LODOP 模板脚本和业务数据，前端不直接执行任意 JavaScript，而是解析允许的命令并调用本地 C-Lodop 服务。该白名单执行思路正确，但仍需确保模板数据仅包含打印所需字段，并对未识别命令明确拒绝。

`PREVIEW` 表示打开预览，`PRINT` 表示向 C-Lodop 发起打印调用。当前系统没有可靠接收和持久化打印机最终结果，因此不能把“调用成功”解释为“物理打印成功”。

## 4. 接口与权限现状

| 接口 | 当前用途 | 当前权限 |
| --- | --- | --- |
| `GET /print-templates` | 查询目标模块可用模板 | 目标模块 `print` 或 `read` |
| `POST /print/items` | 查询单据商品明细 | 目标模块 `read` |
| `POST /print/brands` | 查询品牌数据 | 目标模块 `read` |
| `POST /print/record` | 生成单据打印输出 | 目标模块 `read` |
| `GET/POST /sales-orders/{id}/print-xlsx` | 销售订单 Excel 打印 | `sales-order:print` |
| 打印模板 CRUD | 管理模板 | `print-template` 权限，并叠加目标模块 `update` |

高优先级问题：前端通常只有具备 `print` 权限才显示打印入口，但通用打印数据及输出接口仅要求 `read`。具备读取权限的调用方可以绕过前端直接请求打印接口，造成前后端授权语义不一致。

整改要求：

1. 所有会产生打印输出的通用接口统一校验目标模块 `print` 权限；
2. 数据查询继续执行记录级数据范围校验，不能只校验模块级权限；
3. 模板管理权限与业务打印权限分离，避免模板编辑权限隐式授予业务数据访问；
4. 前端权限只用于交互提示，后端始终作为最终授权边界。

## 5. 模板模型与存储

### 5.1 当前支持模块

运行时配置声明以下 14 个业务模块：

- `purchase-order`、`purchase-inbound`；
- `sales-order`、`sales-outbound`；
- `freight-bill`；
- `purchase-contract`、`sales-contract`；
- `supplier-statement`、`customer-statement`、`freight-statement`；
- `receipt`、`payment`；
- `invoice-receipt`、`invoice-issue`。

配置入口：`src/main/resources/print-runtime.json`。

### 5.2 模板类型

| 模板类型 | 引擎 | 输出方式 |
| --- | --- | --- |
| `COORD` | `LODOP` | 后端装配数据，前端调用 C-Lodop |
| `PDF_FORM` | `PDF_FORM` | 后端 iText 填充 PDF，返回 PDF 输出 |

### 5.3 模板来源

- `FILE`：模板源文件位于 classpath，应用启动时同步到数据库缓存；
- `MANUAL`：用户在线编辑并将模板内容保存到数据库；
- 当前唯一模板持久化表为 `sys_print_template`；
- 当前不存在 `print_job`、`print_job_attempt` 或等价作业表。

数据库脚本 `V3__remove_baseline_print_template_data.sql` 已移除旧基线模板数据，标准模板元数据依赖独立 `db/seed` 版本线。应用若只自动运行主 `db/migration` Flyway，而未运行 seed Flyway，则新环境不会自动获得标准模板。该行为需要在部署流程中明确验证，不应假设模板会随核心 schema 自动出现。

另外，seed 数据中存在 `inventory-report` 模板，但当前运行时模块配置和前端打印入口不支持该模块，属于配置漂移。

## 6. 前端交互现状

### 6.1 “打印作业”命名不准确

弹窗完成的是模板、明细、品牌、单价、备注、排序及预览/打印方式的即时设置，没有创建可查询、可重试的后台作业。建议将界面名称调整为“打印设置”，避免用户误以为关闭弹窗后系统仍会在后台执行。

### 6.2 单据与商品信息

当前通用入口实际按单张业务单据打印，并查询对应商品明细。打印设置和后续历史展示不应只呈现一行单据摘要，还应清晰显示商品名称、规格、材质、品牌、数量、重量、单价、金额等模板真正使用的明细字段。

如果未来引入打印记录页面，推荐采用“单据主行 + 可展开商品明细”的表格，而不是把商品信息压缩成一段文本或仅显示输出文件名。

### 6.3 已发现的交互问题

- `ModuleRecordDetailOverlay.tsx` 中打印按钮没有绑定 `onClick`，上层又固定传入 `canPrint={false}`，属于不可用入口；
- 打印动作缺少完整的 loading 与重复点击保护，可能重复请求并多次打开输出；
- 模板或明细查询失败可能被展示成“无模板”或“无明细”，错误态与空数据态混淆；
- 弹窗关闭后再次打开，没有完整重置隐藏单价、备注、品牌和排序等状态；
- 模板管理中的“预览”主要显示源码，不代表真实业务数据填充后的输出；
- 当前代码实际仅支持单单打印，但仍保留不可达的多记录处理代码，增加理解成本；
- `/print/brands` 及对应前端品牌查询目前未形成有效使用链路，应确认需求后落地或删除。

## 7. 数据与安全风险

### 7.1 打印数据查询范围过宽

`PrintRecordDataProvider` 使用 `SELECT *` 读取记录。特别是在 LODOP 通道中，如果数据未经显式 DTO 白名单投影就返回前端，可能暴露普通业务详情接口不会展示的内部字段。

建议为每个可打印模块定义显式字段投影或打印 DTO：

- 只读取模板允许使用的字段；
- 对供应商、客户、银行账号、内部成本等敏感字段进行权限和场景控制；
- 商品明细采用明确列清单，禁止依赖表结构自然扩展；
- 将记录级数据范围校验放在数据装配前。

### 7.2 缺少业务状态门禁

通用打印链路没有统一限制可打印的单据状态，草稿或业务上尚未生效的单据也可能生成正式样式输出。建议建立模块级打印策略，明确：

- 哪些状态可正式打印；
- 草稿是否只允许带“草稿”水印预览；
- 已作废或已反审核单据是否禁止打印，或必须显示失效标识；
- 重新打印是否需要记录原因。

### 7.3 操作日志语义不准确

后端最多能确认“已生成输出”或“已向客户端返回打印内容”。浏览器打印和本地 C-Lodop 的后续过程不受后端控制，操作日志不能作为物理打印成功凭证。

在没有打印客户端回执前，日志动作应命名为“生成打印输出”或“发起打印”，不要记录为“打印成功”。

## 8. 模板能力缺口

- `is_default` 已持久化但未真正参与默认模板选择；
- `versionNo` 可手工修改，但没有不可变版本历史和回滚能力；
- `assetRef` 可保存，但当前渲染器未消费该字段；
- 未绑定结算主体的模板不一定能作为全局兜底，模板选择规则不清晰；
- 模板管理预览未使用真实渲染链路，无法提前发现字段、分页和坐标问题；
- 前后端及 seed 分别维护支持模块，容易出现 `inventory-report` 一类配置漂移。

建议遵循 YAGNI：对当前业务没有明确用途的字段删除或隐藏；确有用途的能力则补齐闭环测试，避免只保留数据库字段和表单控件。

## 9. 测试现状

审计期间已验证：

- 前端打印相关 22 个 Vitest 文件通过，共 324 个用例；
- 两个核心 Ant Design 打印组件 lint 无问题；
- 未执行真实 Playwright 打印闭环；
- 未连接实体打印机或 C-Lodop 服务验证最终出纸结果。

因此，现有自动化测试可以证明部分前端逻辑稳定，但不能证明浏览器/C-Lodop 集成、模板真实渲染和物理打印成功。

建议至少补充以下场景：

1. 有 `read`、无 `print` 权限时，直接调用打印接口应返回 403；
2. 无记录级访问权限时，即使有模块 `print` 权限也不能打印；
3. 草稿、已审核、已作废等状态分别符合模块打印策略；
4. 查询失败展示错误态，空模板和空明细展示空数据态；
5. 连续点击只能产生一次有效输出请求；
6. 弹窗关闭重开后恢复模块默认设置；
7. PDF 使用真实模板完成字段、分页和下载验证；
8. C-Lodop 使用可控模拟服务验证白名单命令、预览和打印异常处理。

## 10. 近期整改方案

当前业务没有明确提出集中排队、无人值守打印、跨设备打印或失败重试需求，因此近期继续采用同步打印最符合 KISS 和 YAGNI。

### P0：权限与数据边界

1. 将 `/print/items`、`/print/brands`、`/print/record` 等打印链路统一为目标模块 `print` 权限；
2. 为每个模块增加记录级数据范围校验；
3. 将 `SELECT *` 改为打印字段白名单或显式打印 DTO；
4. 增加单据状态门禁，并区分正式打印与草稿预览；
5. 把日志语义改为“生成打印输出”。

### P1：交互与可靠性

1. 将“打印作业”改名为“打印设置”；
2. 修复详情页打印死入口，或在无产品需求时删除入口；
3. 增加 loading、单次提交锁和重复点击保护；
4. 区分加载中、查询失败、无模板和无明细状态；
5. 每次打开弹窗时按当前单据重新初始化全部设置；
6. 打印设置中保留清晰的商品明细表格；
7. 使用真实后端渲染链路实现模板预览。

### P2：配置收敛与无效能力清理

1. 建立单一打印模块注册表，统一模块、权限、模板过滤、结算主体和状态策略；
2. 前端菜单、后端目录、运行时配置和 seed 数据由同一注册信息校验；
3. 落地或删除 `is_default`、`assetRef` 和手工版本号；
4. 清理不可达的多记录代码和未使用的品牌链路；
5. 将 seed Flyway 的执行与校验纳入部署流程。

## 11. 何时升级为真正的打印作业中心

只有出现以下明确业务需求时，才建议引入异步打印作业模型：

- 用户提交后可关闭页面，由后台继续处理；
- 多台打印机集中调度或按仓库、门店路由；
- 大批量打印需要排队、限流和进度展示；
- 打印失败需要自动重试或人工重试；
- 需要追踪客户端、打印机、操作人和最终结果；
- 需要审计重打原因、次数和最终回执。

建议的最小模型：

```text
print_job
  - id
  - module
  - record_id
  - template_id
  - status
  - idempotency_key
  - requested_by
  - requested_at
  - printer_target
  - completed_at

print_job_attempt
  - id
  - job_id
  - attempt_no
  - client_id
  - claimed_at
  - lease_expires_at
  - started_at
  - finished_at
  - result_code
  - error_message
```

推荐状态流转：

```text
PENDING → CLAIMED → PRINTING → SUCCEEDED
                    └───────→ FAILED → PENDING（允许重试）
PENDING / CLAIMED ──────────→ CANCELLED
```

实现该模型时必须同时具备：

- 创建作业幂等键，防止重复点击产生重复作业；
- 带过期时间的领取租约，处理客户端掉线；
- 明确的最大重试次数和人工重试权限；
- 打印客户端完成回执，而不是以前端 HTTP 成功作为最终结果；
- 作业主行和商品明细行的可读历史视图；
- 模板版本快照，保证历史作业可追溯。

若缺少打印客户端回执，建立作业表也只能追踪“输出已领取”，仍不能证明物理打印成功。

## 12. 验收标准

近期同步打印整改完成的最低验收标准：

- 无目标模块 `print` 权限时，所有打印输出接口均拒绝访问；
- 用户只能打印其数据范围内的单据；
- 非允许状态不能生成正式打印输出；
- 打印响应不包含模板白名单之外的业务字段；
- 前端可明确区分加载、错误、空数据和输出中状态；
- 重复点击不会重复生成输出；
- 打印设置重新打开时无上一次单据的残留状态；
- 模板预览与正式输出使用相同渲染链路；
- 日志只记录系统能证明的“生成输出”或“客户端回执”事实；
- 前后端支持模块和 seed 模板不存在配置漂移。

## 13. 架构决策摘要

当前阶段保留同步打印，不引入队列和作业表。优先修复权限、数据投影、状态门禁、交互可靠性和模板配置一致性。只有在集中调度、后台执行、重试或最终结果追踪成为明确需求后，再建设带客户端回执的打印作业中心。
