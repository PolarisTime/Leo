## 2. 结论摘要

本轮修复的核心不是“增加一张退款单”，而是重新明确采购业务的三个不同事实：

1. 采购订单描述合同承诺和原始付款上限，不直接确认最终应付。
2. 采购入库描述实际收货和结算金额，是采购应付的确认依据。
3. 付款与供应商退款到账描述资金结算，分别是结算事件和结算冲回事件。

基于该口径，系统已经形成以下闭环：

- 采购预付款从采购订单产生，并保存供应商稳定编码和采购结算主体快照。
- 采购退款以“采购订单原始值减有效采购入库实际值”计算，未入库数量、重量和金额形成退款差额。
- 供应商可以针对同一采购退款分次退款，只有“已收款”的到账单进入应收应付账簿。
- 供应商退款直接记在对应供应商编码名下，作为应付结算冲回，不再落入无稳定身份的泛化往来方。
- 实际采购入库、采购预付款、供应商退款到账按同一供应商编码和同一结算主体归集；跨主体禁止抵扣。
- 采购、销售、物流和对账下游已增加关键反审核、删除及来源变更保护，避免结算后改写业务事实。
- 物流单和物流对账单核心写链路使用 `carrierCode`，同名不同编码不得合并，编码被业务引用后不得修改。

## 3. 财务事件模型

### 3.1 口径不变量

以下不变量应同时由服务校验、数据库约束和回归测试维护：

1. **确认与结算分离**：出入库确认应收应付，收付款结算应收应付；订单、发票、对账和退款申请不重复制造确认金额。
2. **采购以实际入库为准**：有效采购入库明细的 `amount + weightAdjustmentAmount` 才进入采购应付。
3. **退款以实际到账为准**：采购退款单表达应退差额，供应商退款到账单表达资金已退；只有后者改变结算余额。
4. **稳定身份优先**：供应商使用 `supplierCode`，物流商使用 `carrierCode`，名称只作为快照和展示字段。
5. **结算主体隔离**：账簿分组键至少包括方向、往来方类型、稳定编码、结算主体和对账状态。
6. **状态生效**：草稿、软删除和未达到生效状态的业务单据不得进入有效账簿。
7. **下游不可逆保护**：存在已生效下游关系时，上游不得通过反审核、删除或换来源静默改写历史。
8. **库存与资金退款分离**：未入库差额退款不产生库存出入库流水；库存只由有效采购入库、销售出库等实物单据改变。
9. **账簿完整性不按制单人切割**：`receivable-payable` 是动作权限控制的报表资源，不能按各来源事件的 `created_by`
   过滤，否则同一往来方的确认、结算和退款可能分别被隐藏并产生错误余额。若后续需要组织级财务隔离，应按结算主体或
   独立账簿授权建模，不能复用业务单据的“本人/本部门制单人”范围。

### 3.2 账簿事件表

| 业务事件 | 方向 | 账簿角色 | 金额口径 | 生效条件 | 说明 |
| --- | --- | --- | --- | --- | --- |
| 销售出库单 | 应收 | `RECOGNITION` | 已审核销售出库明细金额汇总 | 未删除且已审核 | 销售订单不重复确认应收 |
| 客户收款核销 | 应收 | `SETTLEMENT` | 分配到已确认客户对账单的金额 | 收款单已收款 | 未分配收款不伪装成已对账核销 |
| 采购入库单 | 应付 | `RECOGNITION` | `amount + weightAdjustmentAmount` 汇总 | 未删除且处于已审核/完成入库等有效状态 | 采购订单原始金额不直接进入应付确认 |
| 普通供应商付款 | 应付 | `SETTLEMENT` | 分配到已确认供应商对账单的金额 | 付款单已付款 | 仅核销匹配的供应商和结算主体 |
| 采购预付款 | 应付 | `SETTLEMENT` | 已付款金额；按已分配与未分配部分拆分 | 用途为 `PURCHASE_PREPAYMENT` 且已付款 | 已分配部分进入已对账，剩余部分保留为未对账结算 |
| 采购退款单 | 应付 | 不直接入账 | 订单原始值减有效入库实际值 | 业务单据审核 | 只表达退款权利，避免与预付款、入库和到账重复记账 |
| 供应商退款到账单 | 应付 | `SETTLEMENT_REVERSAL` | 本次实际到账金额 | 未删除且已收款 | 作为负的净结算额，直接归入供应商名下 |
| 物流单 | 应付 | `RECOGNITION` | 已审核物流单运费 | 未删除且已审核 | 使用物流商身份和物流结算主体分组 |
| 物流付款核销 | 应付 | `SETTLEMENT` | 分配到已审核物流对账单的金额 | 付款单已付款 | 与供应商付款保持不同往来方类型 |
| 台账调整单 | 应收/应付 | `RECOGNITION` 或 `SETTLEMENT` | 由方向和“增加/减少余额”共同决定借贷方向 | 未删除且已审核 | V17 增加结算主体，防止跨主体调整 |

开票、收票和对账单仍保留来源、状态、稳定编码和结算主体，用于证据链、容量校验与核销分组；它们不应再次复制
销售出库或采购入库已经确认的应收应付金额。

### 3.3 采购退款示例

以现有 PostgreSQL 回归场景为例：

| 事件 | 金额 | 对应账簿影响 |
| --- | ---: | --- |
| 采购订单原始付款基数 | 117000.00 | 不直接确认应付 |
| 实际有效采购入库 | 114523.50 | 确认应付 114523.50 |
| 采购预付款已支付 | 117000.00 | 结算应付 117000.00 |
| 采购退款差额 | 2476.50 | 退款业务依据，不直接重复入账 |
| 供应商退款实际到账 | 2476.50 | 冲回结算 2476.50 |

最终 `recognizedAmount = 114523.50`，`settledAmount = 117000.00 - 2476.50 = 114523.50`，余额为 0。
草稿退款到账单不会参与上述计算。

## 4. 采购退款与预付款闭环

### 4.1 采购预付款

付款单通过 `paymentPurpose` 区分普通对账付款和采购预付款：

- `STATEMENT_SETTLEMENT`：按既有方式关联对账单。
- `PURCHASE_PREPAYMENT`：关联采购订单，不能在创建时携带对账核销明细。

采购预付款来源必须是未删除且处于“已审核”或“完成采购”的采购订单。后端从来源订单锁定并复制：

- `sourcePurchaseOrderId`、采购订单号；
- `supplierCode`、供应商名称；
- `settlementCompanyId`、结算主体名称；
- 供应商编码同时写入付款单往来方编码，避免只按名称归集。

同一订单允许按业务需要分次预付，但累计“已付款”金额不得超过按订单明细理论重量和单价计算的原始金额。
前端使用 `/purchase-orders/prepayment-candidates` 专用分页，不复用普通采购订单列表再做本地过滤。

付款后可将采购预付款分配给供应商对账单。替换分配明细时，后端锁定付款根记录和新旧对账来源，并再次校验：

1. 付款用途确为采购预付款且状态为已付款。
2. 对账单已确认。
3. 对账单供应商编码与预付款一致。
4. 对账单结算主体与预付款一致。
5. 分配合计不超过预付款金额。

### 4.2 采购退款计算

`PurchaseRefundCalculator` 按采购订单明细逐行计算：

- 退款数量 = `max(采购订单数量 - 有效入库数量合计, 0)`；
- 退款重量 = `max(订单理论重量 - 有效入库实际/过磅重量合计, 0)`；
- 退款金额 = `max(订单理论重量 × 单价 - 有效入库(amount + weightAdjustmentAmount)合计, 0)`。

有效入库仅包含未软删除且处于“已审核”或“完成入库”的采购入库。退款预览和最终保存均由后端重算，
前端不能提交自定义退款明细覆盖该结果。审核采购退款前还必须满足：

- 来源采购订单处于允许状态；
- 来源订单采购预付款已经足额支付；
- 差额至少在数量、重量或金额之一为正；
- 同一采购订单不存在另一张活动采购退款单；
- 供应商稳定编码和结算主体快照完整；
- 已收票金额不会因退款后可开票容量收缩而超过允许值。

这里的采购退款是“已付款但未入库部分的资金退款”，不是“已入库商品退货”。因此采购退款单不写库存，也不会为未入库商品
制造负库存或虚假出库。若未来需要把已经入库的实物退回供应商，应单独设计采购退货出库及其库存成本冲回，不能复用本单据。

### 4.3 供应商退款到账

供应商退款到账单直接引用已审核采购退款单，并从来源复制供应商编码、名称和结算主体。允许同一退款单分次到账，
但所有未删除到账单的累计金额不得超过采购退款总额。容量校验在锁定来源订单明细和退款单后执行，避免并发分次到账超额。

状态规则：

- 草稿：可编辑，不进入应收应付账簿。
- 已收款：形成供应商名下的 `SETTLEMENT_REVERSAL` 事件。
- 已存在到账关系后，采购退款和来源预付款的关键生命周期变更会被阻止，防止历史资金事实失去来源。

## 5. 稳定身份与结算主体隔离

### 5.1 供应商身份

V14 为采购订单和采购入库增加 `supplier_code` 快照；V15 为收票单增加供应商编码和结算主体快照；V16 将同一身份
继续传递到采购预付款和供应商退款到账。回填只接受可唯一确定的历史名称，无法唯一匹配时迁移失败，不会任取一条主数据。

供应商编码被采购、收票、预付款或退款业务引用后，主数据服务禁止直接修改编码。名称仍可作为历史展示快照变化，
但汇总键不随同名或改名而漂移。

### 5.2 物流商身份

V18 为物流单增加非空 `carrier_code`，并要求物流对账单从全部来源物流单继承同一个编码：

- 同名不同编码不能合并生成物流对账单；
- 请求、响应、选项和分页筛选均传递编码；
- 物流商编码被物流单或物流对账单引用后禁止修改；
- 物流业务核心写链路已移除“按名称取第一条”回退。

由于 `md_carrier.carrier_code` 使用“活动记录部分唯一索引”，PostgreSQL 不能把它作为普通外键的被引用唯一键，
V18 没有添加普通外键。当前完整性边界由非空/非空白约束、服务端解析和编码生命周期保护共同承担。

应收应付的物流单确认分支已经直接以 `bill.carrier_code` 作为往来方编码和分组键，不再连接名称映射；真实 PostgreSQL
红绿测试覆盖了两个同名不同编码物流商，证明物流单会归入自身编码而不会被错误归到排序靠前的编码。

### 5.3 结算主体

应收应付 SQL 使用 `settlement_company_id` 构造独立分组键；同一供应商或物流商在不同结算主体下会产生不同账簿。
预付款分配、收付款核销、供应商退款、入库确认、对账及台账调整都必须保留并校验该维度。

这意味着以下操作必须拒绝：

- 用 A 结算主体的预付款核销 B 结算主体的供应商对账单；
- 把不同结算主体的采购入库合并成一张供应商对账单；
- 把不同物流结算主体的物流单合并成一张物流对账单；
- 通过台账调整单把金额写入无对应结算主体的汇总组。

## 6. 下游变更保护

| 上游对象 | 已识别的下游事实 | 保护行为 |
| --- | --- | --- |
| 采购订单 | 采购预付款、采购退款、供应商退款到账、收票来源 | 锁定新旧来源；存在生效下游时禁止删除、换来源或破坏退款生命周期的修改 |
| 采购入库 | 已审核采购退款、供应商对账、付款/收票关联 | 退款已审核后禁止跨越审核边界；删除和来源变化继续执行既有下游保护 |
| 销售订单 | 已审核销售出库、客户对账、收款和开票来源 | 已存在生效下游时禁止反审核、删除或修改关键来源事实 |
| 销售出库 | 客户对账、客户收款、物流单 | 已生成任一相关下游后禁止反审核；删除和来源修改执行相同事实保护 |
| 物流单 | 物流对账及其付款核销 | 已生成物流对账后禁止反审核和删除；已付款时返回更明确的结算阻断原因 |
| 客户/供应商/物流对账单 | 已收款或已付款核销 | 禁止反审核、删除及修改往来单位或来源，必须先反核销对应收付款单 |
| 开票/收票来源 | 已审核发票及其来源容量 | 上游变更不得使既有发票来源失效或超出容量 |

这些保护不能只依赖页面按钮禁用。服务端必须在同一事务内锁定来源、重新读取下游状态并失败闭合，以覆盖并发请求、
旧客户端和直接 API 调用。

该指纹仅用于人工比对，最终必须使用目标环境的 `flyway_schema_history` 和 `flyway:validate` 核对 Flyway checksum。
如果内容不同，严禁覆盖 V12，只能新增更高版本迁移修复。V12 至 V18 在部署前必须全部进入同一受控提交并冻结内容。

### 7.2 V18 上线风险

V18 的失败闭合场景包括：

1. 一个有效物流商名称对应多个不同编码。
2. 历史物流单无法匹配唯一有效物流商编码。
3. 同一物流对账单来源物流单包含多个物流商编码。
4. 物流对账单已有编码与来源物流单编码冲突。
5. 无来源的历史物流对账单也无法按唯一名称补齐编码。

任一场景都会抛出异常并回滚整条迁移。该行为用于防止猜测式修复身份，不能通过放宽 SQL 或临时写入任意编码绕过。

此外，回填、`NOT NULL`、检查约束和索引会扫描相关表并取得锁。数据量较大时必须在生产数据副本上测量执行时间、
锁等待和额外磁盘空间，再安排维护窗口。V18 成功后旧版本服务写入物流单时不会提供 `carrier_code`，插入将失败；
因此必须停旧写入、执行迁移、部署新服务并完成冒烟后再恢复流量，禁止滚动期间新旧版本混合写入。

## 8. 发布前预检与部署步骤

### 8.1 迁移前

1. 只读核对开发库和生产库各自的 `flyway_schema_history`，确认 V12 的实际状态以及不存在失败、乱序或 checksum 异常。
2. 在生产数据副本执行 V13 至 V18 的 `flyway:migrate` 和 `flyway:validate`，记录每条迁移耗时与锁等待。
3. 预检采购订单、采购入库和收票单的供应商名称是否能唯一映射编码；发现同名多编码时由业务确认正确身份。
4. 预检物流单和物流对账单的名称唯一性、来源完整性及编码冲突；所有异常必须形成显式修复清单。
5. 统计受影响表行数与索引空间，设置可控的维护窗口、备份点和应用停写方案。
6. 保持后端最新 6020 项全量单测和前端全量门禁通过；后续代码若继续变更，必须重跑对应门禁。不得使用 2026-07-10 V1 至 V11 批次或本轮变更前 5995 项的旧测试数量替代最新结果。

### 8.2 发布顺序

1. 停止会写入采购、付款、退款、物流和对账相关表的旧版本流量。
2. 备份并确认可恢复点。
3. 执行 Flyway `migrate`，随后立即执行 `validate`。
4. 部署与 V18 契约匹配的新后端，再部署包含 `carrierCode`、采购退款和预付款页面的新前端。
5. 执行创建、编辑、审核、反审核、分页筛选、预付款分配和分次退款到账冒烟。
6. 恢复流量并监控业务异常码、Flyway 日志、慢 SQL、锁等待和账簿余额变化。

### 8.3 回滚边界

- 迁移执行中失败：依赖 PostgreSQL 事务回滚，并在修复数据后重新执行；禁止手工标记成功。
- V18 已成功但应用冒烟失败：不能直接恢复会写缺失 `carrier_code` 的旧服务；应优先前滚修复，或先提供兼容写入版本。
- 已产生 V16 新业务数据后：不得通过删除表/列回滚，应以更高版本 Flyway 和兼容应用进行前滚修复。

## 10. 剩余风险与审计边界

### 10.1 发布前必须处理或确认

1. **当前后端测试与 Verify 已通过，但仍不是完整发布结论**：6020 项单测以及 Checkstyle、打包、JaCoCo 已通过；SpotBugs 仍有 51 个既有 High 告警，且共享环境 Flyway validate、生产副本迁移和应用冒烟尚未完成。
2. **V12 checksum 尚未由 Git 历史证明**：V12 已在共享开发库执行，但审计时迁移文件尚未纳入 Git；必须先用该环境的 Flyway history/validate 核对，任何差异都只能通过更高版本修复。
3. **生产历史数据尚未预检**：V14、V15、V18 都会因身份无法唯一回填而失败，必须提前发现而不是在发布窗口临时处理。
4. **采购退款候选仍为批量扫描后内存分页**：当前每批读取 200 张采购订单再计算可退差额；大数据量下需改为数据库候选分页或可索引的退款进度模型。
5. **普通付款历史兼容分支仍有末级名称回退**：物流单确认分支已直接使用 `bill.carrier_code`，物流单识别链路已经闭环；普通对账付款分支则按“对账单稳定编码 -> 付款单往来方编码 -> 名称映射”顺序兼容历史数据。前两级覆盖新数据，名称映射只应作为旧数据兜底；上线前需确认历史付款都能命中稳定编码，并制定清理该兜底的迁移计划，避免同名多编码被任取排序靠前的编码。
6. **V18 没有普通外键**：这是部分唯一索引的数据库限制，不是可以忽略完整性的理由；服务校验、非空约束和编码生命周期测试必须持续保留。

## 11. 最低验收标准

1. 一张足额预付但部分入库的采购订单，退款预览严格等于订单原始值减有效入库实际值。
2. 退款审核前预付款未足额时失败；足额后成功；已经分次到账时累计金额永不超过退款金额。
3. 采购退款单审核本身不重复改变应付，供应商退款到账后才以冲回事件改变净结算额。
4. 同一供应商、不同结算主体在列表、明细、导出和账龄中始终为独立分组。
5. 同名不同编码供应商或物流商不会被合并；任何无法确定唯一编码的数据都失败闭合。
6. 已发生收付款、退款到账或下游对账后，上游反审核、删除和来源变更请求被服务端拒绝。
7. 测试库可从 V1 顺序迁移至 V20，`flyway:validate` 无失败；共享开发库和生产库按受控流程升级且无混合版本写入。
8. 后端 6020 项全量单测保持通过，并清理或受控基线化 SpotBugs 告警；前端已通过的全量单测、静态检查和生产构建在后续代码发生变化时必须重跑。
